## INT8 variants: do calibration methods change fusion?

If three INT8 engines (built with different calibration methods
but the same calibration data size) show identical layer counts
and fusion counts, then mAP differences observed between them
are attributable to quantization SCALES alone — not to the
compiler choosing different fusion/kernel paths.

### Verdict

**Fusion decisions are identical across the three INT8 variants.** Stages compared: After scale fusion, After vertical fusions, After slice removal. Fusion-type counts are also identical.

Post-fusion graph cleanup differs slightly across variants (concat removal / reformat layers). This is a byproduct of how quant/dequant nodes are resolved during the calibration step (entropy/minmax went through the 2-phase calibrator API; percentile used a cache-injection 1-phase path) and is **unrelated to fusion decisions**.

Together with §4.2 (per-tensor quantization scales differ by up to 8× across calibration methods), this isolates the mAP gap between minmax (0.380) and entropy/percentile (0.266–0.269) to quantization scales alone — not the compiler choosing different fusion or kernel paths.

## Observations

1. **Vertical fusion is the dominant size reduction.** 248 layers → FP32 179 → FP16 129 → INT8 124. Higher quantization enables more vertical fusion, not less. INT8 ends up smaller than FP16, contrary to a common assumption that quantization fragments the graph.

2. **INT8 builds run a calibration phase first.** Phase 1 leaves the network at 248 layers (vertical fusion skipped) but inserts quant/dequant nodes (concat removal grows to ~279). Phase 2 then applies full fusion with the calibrated scales.

3. **Reformat layers track precision boundaries.** FP16 has 152 layers after reformat, INT8 has 133. Reformats are dtype-conversion stubs inserted where a layer's input/output precision differs from the surrounding graph. INT8 needing fewer reformats than FP16 is one structural reason its kernel-level runtime is faster.

4. **Calibration method does not affect fusion.** The three INT8 engines (entropy / minmax / percentile) produce identical fusion decisions at every fusion stage (vertical fusions, slice removal) and identical fusion-type counts. Post-fusion graph cleanup (concat removal, reformat layers) differs by 2-5 layers across variants, but that reflects how quant/dequant nodes are resolved during calibration (a 2-phase calibrator API vs a 1-phase cache-injection path) — not the compiler picking different fusion paths. Combined with §4.2 (quantization scales differ by up to 8× across methods), this isolates the mAP gap to quantization scales alone.

5. **Fusion type distribution is precision-invariant.** PointWiseFusion fires the same number of times (65) across FP16 and all INT8 variants. GenericConvActFusion fires slightly more in INT8 (50 vs 44 in FP16), suggesting that quantized Conv outputs interact more uniformly with the following activation.
